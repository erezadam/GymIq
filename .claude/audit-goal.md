## מבנה עבודה

בצע את 4 התחומים כ-subagents נפרדים בתוך אותו session אחד (Task tool /
subagent פנימי), במקביל אם אפשר. אל תפתח sessions נפרדים של Claude Code ואל
תשתמש ב-git worktree נוסף — המשימה read-only ולכן רצה כולה בתהליך אחד על
עותק העבודה הקיים. כל subagent מחזיר ראיות גולמיות (תוכן קובץ, פלט פקודה,
diff רלוונטי) ל-agent הראשי, שמאחד הכל לדוח סופי אחד.

לפני שמתחילים: ודא שאתה בתיקיית הפרויקט הנכונה (`pwd` + `git remote -v`),
ושה-working tree נקי (`git status`). אם יש שינויים לא-commit-ים — עצור ודווח,
אל תמשיך.

## תחום 2: ביצועים ומהירות טעינה

1. הרץ build עם ניתוח bundle size (`vite build`; אם יש plugin ניתוח מובנה —
   השתמש בו, אחרת דווח את גדלי ה-chunks מפלט הבנייה הרגיל). ציין את 10
   ה-chunks הכי גדולים.
2. בדוק את `App.tsx` — כל route שאינו lazy-loaded, ציין אותו במפורש עם הסבר
   למה זה משפיע על טעינה ראשונית.
3. בדוק את `useActiveWorkout.ts` (ההוק הכי מורכב) לבעיות re-render: state
   updates לא ממוזערים, dependencies שגויים ב-useEffect, אובייקטים/מערכים
   שנוצרים מחדש בכל render ומועברים כ-props.
4. סרוק שאילתות Firestore בכל `src/lib/firebase/*.ts` לדפוסי N+1 (לולאה עם
   getDoc בפנים במקום query מרוכז) והשווה לאינדקסים הקיימים
   ב-`firestore.indexes.json` — האם יש שאילתות שרצות בלי אינדקס תואם?
5. בדוק את `public/sw.js` — מה כן/לא מטמון, והאם אסטרטגיית ה-cache הגיונית
   לתמונות תרגילים ול-webp.
6. בדוק שימוש ב-TanStack Query מול fetch ידני ב-useEffect (לפי CODEATLAS
   סעיף 10) — אילו מסכים בעלי טעינת נתונים כבדה לא משתמשים ב-caching בכלל
   ויכולים להרוויח ממנו.
7. חפש `console.log` שנשארו בקוד production (לא רק ב-`scripts/`).
